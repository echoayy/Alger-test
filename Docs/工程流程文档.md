# 壳木AI · Unity 3D 场景游戏工程流程文档

> 策略：**基础设施层固定，玩法层留空**
> 每次收到新需求，按本文档的阶段顺序逐步实现，不跨步。

---

## 一、目录结构规范

```
壳木AI/
└── Assets/
    ├── Starter/                  ← 基础设施层（稳定，轻易不改）
    │   ├── Core/                 # 核心系统：事件总线、单例基类、服务定位器
    │   ├── Bootstrap/            # 启动流程：场景加载器、初始化序列
    │   ├── Runtime/              # 运行时工具：对象池、计时器、协程管理
    │   ├── Authoring/            # 编辑器扩展：自定义Inspector、Gizmos辅助
    │   ├── UI/                   # UI基础框架：面板管理、弹窗系统
    │   ├── Debug/                # 调试工具：日志、性能监控、作弊码
    │   ├── Scenes/               # 基础场景：Bootstrap场景、Loading场景
    │   ├── Resources/            # 全局资源：字体、公共材质、图标
    │   └── Gizmos/               # 编辑器图标资源
    │
    └── Game/                     ← 玩法层（每个需求在此扩展）
        ├── Scripts/              # 游戏逻辑脚本
        │   ├── Player/           # 玩家相关
        │   ├── Enemy/            # 敌人/NPC
        │   ├── World/            # 世界/场景逻辑
        │   ├── System/           # 游戏系统（战斗、任务、背包…）
        │   └── Data/             # ScriptableObject 数据定义
        ├── Prefabs/              # 预制体
        │   ├── Player/
        │   ├── Enemy/
        │   ├── Environment/
        │   └── VFX/
        ├── Scenes/               # 游戏场景文件
        ├── UI/                   # 游戏UI（依赖Starter/UI框架）
        ├── Audio/                # 音效与音乐
        ├── Art/                  # 美术资源（模型/贴图/动画）
        │   ├── Models/
        │   ├── Textures/
        │   ├── Materials/
        │   └── Animations/
        └── Config/               # 游戏配置（ScriptableObject实例）
```

## 二、分层架构说明

| 层级 | 目录 | 职责 | 可依赖 |
|------|------|------|--------|
| 基础设施层 | `Starter/` | 与游戏无关的通用框架 | 仅Unity API |
| 玩法层 | `Game/` | 具体游戏逻辑 | 可依赖 Starter |
| 数据层 | `Game/Data/` `Game/Config/` | ScriptableObject配置 | 不依赖逻辑脚本 |

**规则：** `Starter/` 的代码不得引用 `Game/` 的任何类型。

## 三、功能开发标准流程

收到一个新需求时，按以下 **6个步骤** 顺序执行：

### Step 1 · 需求拆解
```
□ 用一句话描述这个功能的核心行为
□ 列出涉及的游戏对象（角色/场景物件/UI）
□ 列出输入（触发条件）和输出（结果/反馈）
□ 确认是否需要持久化（存档）
```

### Step 2 · 数据定义（Data/Config）
```
□ 为功能创建 ScriptableObject 数据类（如有配置需求）
   路径：Assets/Game/Data/Xxx定义.cs
   实例：Assets/Game/Config/Xxx配置.asset
□ 在数据类中只放纯数据字段，不写逻辑
```

### Step 3 · 目录与脚本骨架
```
□ 在 Game/Scripts/ 对应子目录下新建脚本文件
□ 先写类的骨架：字段声明 + 空方法签名 + 注释说明
□ 在 Game/Prefabs/ 下准备对应预制体（可先用空GameObject占位）
```

### Step 4 · 核心逻辑实现
```
□ 实现 Step 3 中各空方法的具体逻辑
□ 事件通信优先走 Starter/Core/ 的事件总线，不直接跨模块GetComponent
□ 需要对象复用的走 Starter/Runtime/ 的对象池
□ 每个方法控制在 20 行以内；超过则拆分
```

### Step 5 · 场景集成与测试
```
□ 在对应场景中挂载脚本/预制体
□ 在 Unity Editor 中 Play Mode 验证主流程
□ 验证边界情况（快速连点、极值输入、场景切换后状态）
□ 检查 Console 无报错、无警告
```

### Step 6 · 收尾
```
□ 清理调试用的 Debug.Log
□ 为公开字段填写 Tooltip
□ git add + git commit（提交信息格式见第五节）
```

## 四、3D 场景开发专项规范

### 4.1 场景结构（Hierarchy 层级）
```
[Scene Root]
├── --Environment--
│   ├── Terrain
│   ├── StaticMeshes
│   └── Lighting
├── --Gameplay--
│   ├── Player
│   ├── Enemies
│   └── Interactables
├── --UI--
│   └── Canvas (Screen Space - Overlay)
├── --Managers--          ← 不可见的管理器对象
│   ├── GameManager
│   ├── AudioManager
│   └── CameraManager
└── --Debug--             ← 仅编辑器可见
```

### 4.2 相机系统
```
□ 使用 Cinemachine Virtual Camera 管理镜头
□ 镜头切换通过优先级(Priority)控制，不手动开关 Camera
□ 所有镜头预制体放在 Game/Prefabs/Camera/
```

### 4.3 物理与碰撞规范
| Layer | 用途 |
|-------|------|
| Default | 静态环境 |
| Player | 玩家 |
| Enemy | 敌人 |
| Interactable | 可交互物体 |
| Projectile | 子弹/投射物 |
| UI | UI遮挡检测 |

### 4.4 性能基准
```
□ 同屏 Draw Call < 200（移动端 < 100）
□ 静态物体勾选 Static → Batching
□ 距离 > 50 单位的物体启用 LOD
□ 粒子系统最大粒子数上限 500
```

## 五、命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 类/组件 | PascalCase | `PlayerController` |
| 方法 | PascalCase | `TakeDamage()` |
| 私有字段 | _camelCase | `_currentHealth` |
| 公开属性 | PascalCase | `MaxHealth` |
| 常量 | UPPER_SNAKE | `MAX_JUMP_COUNT` |
| 场景文件 | PascalCase | `MainWorld.unity` |
| 预制体 | PascalCase | `EnemyGoblin.prefab` |
| ScriptableObject | XxxData / XxxConfig | `EnemyData.asset` |

## 六、Git 提交规范

```
<类型>(<范围>): <简短描述>

类型：
  feat     新功能
  fix      Bug修复
  refactor 重构（不改变行为）
  art      美术/场景资源
  config   配置调整
  docs     文档

示例：
  feat(Player): 添加二段跳逻辑
  fix(Enemy): 修复巡逻状态下NPC卡墙问题
  art(World): 更新主场景地形贴图
```

## 七、功能需求接收模板

> 每次告知新需求时，可按此格式描述，方便精准拆解：

```
【功能名称】
【触发方式】玩家按下X键 / 进入某区域 / 时间到达…
【期望行为】主角做了什么 / 场景发生了什么变化
【涉及对象】哪些GameObject/系统会受到影响
【特殊要求】性能限制 / 平台限制 / 视觉风格要求
```
